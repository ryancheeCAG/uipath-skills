Dashboard Deployment

From localhost to Automation Cloud

Phase 1 — Already done
Local development
The build script runs locally. Your dashboard is live at http://localhost:57173 for you to test and adjust. All widgets, charts, and data are working against your real UiPath tenant. This phase is complete before you deploy.
Phase 2 — Deploy
Production on Automation Cloud
Your local project gets built, packaged, and published to Automation Cloud as a Coded Web App. The result is a permanent URL your team can access without running anything locally. UiPath handles authentication — no OAuth app needed for team members.
🔗 state.json — the bridge between build and deploy
The build script writes .dashboard/state.json after every successful build. The deploy plugin reads it — no re-asking for what dashboard this is, what version, or what URL slug to use.
routingName — the permanent URL slug (e.g. agent-health-x7k2). Never changes.
semver — current version. Deploy bumps the patch number automatically.
deployment.systemName — set after first deploy. Used to detect "upgrade" vs "fresh".
deployment.appUrl — the live cloud URL, updated after each deploy.
The 10-step deploy flow
1
User action
User says "deploy this dashboard"
Triggers the deploy plugin. The agent reads state.json to understand the current state — app name, version, whether it's been deployed before, and which cloud folder it lives in (if already deployed).
2
Automatic
Classify: fresh deploy or upgrade
If deployment.systemName is null → first time this dashboard is deployed → fresh deploy.
If deployment.systemName is set → updating an existing app → upgrade (uses the existing deployment slot).
3
User action (fresh deploy only)
Choose a folder
For a first deploy, the agent lists your Orchestrator folders and asks which one to deploy to. Your answer is saved in state.json so future upgrades go to the same folder automatically.
4
Automatic
Version bump + conflict check
Increments the patch version (1.0.0 → 1.0.1). Checks if that version is already published — if so, bumps again. No manual versioning needed.
5
User action
Confirm the deploy plan
Agent shows a clear summary and waits for confirmation before touching anything in the cloud.
Agent Health Dashboard · 1.0.0 → 1.0.1 Fresh deploy to "My Automation Folder" URL slug: agent-health-x7k2
6
Automatic
Production build
Runs npm run build (Vite) to produce an optimised static bundle in dist/. The dev credentials file (.env.local) is temporarily moved out so it never enters the production bundle — a Vite plugin enforces this and will fail the build if a PAT key is present.
7
Automatic
Pack
Bundles the dist/ output into a deployable package using the routing slug as the package identifier. This is the same slug that becomes part of the permanent URL.
8
Automatic + self-healing
Publish to the platform registry
Uploads the package to the Automation Cloud registry. Handles two failure modes automatically:
· 409 conflict (version already exists) → bumps version and retries
· 5xx / gateway errors → waits briefly and retries up to 4 times
9
Automatic + self-healing
Deploy the app to the folder
Makes the dashboard accessible to users in the chosen Orchestrator folder. Also handles URL slug conflicts — if the routing name is already taken in that folder, it generates a new suffix and retries (up to 3 times).
10
Automatic
Save and report
Updates state.json with the new version, system name, and live URL. The agent then shows the URL and tells the user how to update it in the future.
