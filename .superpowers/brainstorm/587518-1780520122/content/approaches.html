Three architectural approaches to robust dashboard generation

Each makes a different bet on where intelligence and complexity should live

A

Approach A — Optimised Instruction-Driven

What it is: Improve the current 7-phase approach. Better pre-warming, cleaner plan format, progress updates during build. The AI agent still orchestrates every phase.

User NLP → Agent derives full widget config → Agent writes plan.json → build script executes

Pros

  • Least new work — incremental
  • Agent flexibility for unusual requests

Cons

  • Agent still writes TypeScript config — hallucination risk
  • 7 phases = 7 failure points
  • Hard to make faster without structural change

Speed: 90–150s typical. Hard floor at ~60s due to npm install.

B

Approach B — Code-as-Orchestrator ⭐ Recommended

What it is: The build script becomes the brain. The agent's only job is NLP → a simple intent JSON (metric names + time range). All metric routing, template selection, widget config generation, and code writing happens in deterministic Node.js code — not the AI.

User NLP → Agent writes 10-line intent.json → build-dashboard.mjs does everything else
Intent JSON is tiny and safe:
{ "name": "Agent Health", "metrics": ["agent-errors", "invocation-volume", "top-failing-agents"], "timeRange": "30d" }

Pros

  • Agent can't hallucinate widget TypeScript
  • Build script is testable, versioned, deterministic
  • Progressive disclosure is natural (stream build phases)
  • Easy to add new metric types without skill changes
  • Agent failure is contained — code handles the hard part

Cons

  • Larger build script to maintain
  • Unusual/custom metrics need extension points

Speed: 30–60s for standard metrics. Pre-warm hidden behind plan. Progressive: first widget visible in browser while rest build.

C

Approach C — Catalog + NLP Customisation

What it is: 6–8 fully pre-compiled dashboard bundles (Agent Health, RPA Operations, Governance, Job Performance, etc.). Agent maps NLP to the nearest catalog entry and applies deltas (add/remove widgets). Common case: zero npm install (bundle already built).

User NLP → Agent matches catalog entry → Apply customisation delta → Dev server (pre-built)

Pros

  • Fastest possible — catalog dashboards open in <15s
  • Zero AI-generated code in hot path
  • Perfect for common use cases

Cons

  • Catalog must be maintained as APIs evolve
  • Custom requests fall back to Approach B anyway
  • Binary files in repo (pre-built bundles)

Speed: <15s for catalog match. Falls back to 30–60s for custom. Best average case, worst maintenance burden.

My recommendation: B, with C as a future optimisation

Approach B gives the best robustness/speed balance without maintaining a catalog. Once B is proven, adding a catalog layer (C) for the 6 most common dashboards is a straightforward optimisation — the catalog entries are just pre-run outputs of B.
