Sections 4 & 5: Progressive Build + Incremental Editing

How the build stays fast and how users keep shaping the dashboard after it's running

Section 4 — Progressive Build & Pre-warm Guarantee

Pre-warm — guaranteed, not timed

Current problem: pre-warm fires and build continues regardless — if npm is slow, build-dashboard.mjs fails mid-flight.

New design: build script writes a .prewarm.lock file when npm ci starts. Code generation phase polls for node_modules/.package-lock.json before writing any widget file.

If pre-warm hasn't finished when user confirms: build script shows "Installing dependencies (5s)..." — still inside build phase, not a failure.

Result: pre-warm is decoupled from plan review timing. Works even if user confirms instantly.

Event streaming protocol

PREWARM_DONE
SCAFFOLD_READY
ENV_WRITTEN
WIDGET_READY:{"name":"ErrorRateTrend","index":1,"total":4}
WIDGET_READY:{"name":"InvocationVolume","index":2,"total":4}
T3_RETRY:{"widget":"CustomQuery","errors":["TS2339: ..."]}
WIDGET_READY:{"name":"CustomQuery","index":3,"total":4}
WIDGET_READY:{"name":"HighFailureQueues","index":4,"total":4}
TSC_PASS
SERVER_READY:{"port":5173,"url":"http://localhost:5173"}
BUILD_RESULT:{"success":true,"widgets":[...]}
Agent parses stdout line-by-line. Each WIDGET_READY line prints a progress tick. T3_RETRY triggers agent to rewrite fn body — build script reads the fix from stdin and continues.

Widget generation is parallel inside the build script

T1 and T2 widgets generate in parallel (Promise.all). T3 widgets run sequentially because they require agent interaction (T3_RETRY round-trips). A dashboard with 4 widgets where 3 are T1/T2 takes the time of the slowest T1/T2 widget (~1s) plus T3 resolution time (~5–10s) — not 4× sequential.

Section 5 — Incremental Editing

Classification — what the agent determines

type EditIntent =
 | { op: "ADD"; metric: MetricRef }
 | { op: "REMOVE"; target: string }  // widget name
 | { op: "CHANGE"; target: string; delta: Partial<MetricRef> }
 | { op: "REBUILD" }  // full regeneration
Agent writes a single edit-intent.json. Build script reads state.json, applies the diff, regenerates only affected files, hot-reload fires automatically.

Hand-edit protection

Build script hashes each generated widget file at write time and stores in state.json:

"ErrorRateTrend": { "hash": "a3f7...", "tier": "T1" }

On CHANGE/REMOVE: if current file hash ≠ stored hash, the file was hand-edited. Build script emits HAND_EDIT_DETECTED event — agent warns user before overwriting.

REBUILD always asks for confirmation before overwriting any hand-edited file.

Routing name is permanent

Derived at first build from dashboard name + 4-char random suffix. Stored in state.json. Never changes on edits, never changes on deploy upgrades. This ensures the app's URL slug and Orchestrator package identity remain stable across the dashboard's lifetime.

New session resume

On fresh session start, agent checks for .dashboard/state.json. If found:

1. Reads widget list, org, tenant, routing name
2. Starts dev server (no rebuild needed)
3. Reports: "Found existing dashboard with N widgets. Server running at http://localhost:5173. What would you like to change?"

Zero re-prompting. Zero re-setup.
