How does the user trigger dashboard generation?

This is the single biggest architectural decision — it determines latency, complexity, and how much guidance the user needs

A

Natural language → done

User types one sentence: "Show me agent error rates and invocation volume for the last 30 days." Claude interprets intent, picks widgets, builds, and opens a preview URL — no plan review, no back-and-forth. Sub-60 second target.

Pros

  • Fastest experience
  • Zero decisions from user

Cons

  • User may not know what data exists
  • First attempt may miss intent
B

Natural language → quick plan → build

User types intent. Claude shows a 10-second summary ("I'll build 4 widgets: Error Rate, Invocation Volume, Top Failing Agents, Active Jobs — confirm?"). One confirmation, then builds. 90–120 second target.

Pros

  • User stays in control
  • Catches misunderstandings early

Cons

  • One extra step
  • User must understand widget names
C

Named dashboard catalog

A curated set of pre-defined dashboards: "Agent Health", "RPA Operations", "Governance & Compliance", "Job Performance". User picks a name, optionally customises time range. Deterministic, fast, zero ambiguity.

Pros

  • Fully deterministic
  • Best performance (templates preloaded)
  • No hallucination risk

Cons

  • Limited to what's in catalog
  • Requires maintaining catalog
D

Catalog + natural language customisation

Start with named templates (Option C) for speed and reliability. Add a customisation layer — user can say "same as Agent Health but also show memory usage." Combines determinism with flexibility.

Pros

  • Best of both worlds
  • Fast common path, flexible edge cases

Cons

  • More complex to build
  • Catalog maintenance needed
