Section 6: Error Handling & Resilience

Every failure mode has a defined recovery path. Nothing silently produces broken output.

Tier 3 tsc failure → retry loop

Signal: T3_RETRY:{"errors":["TS2339: Property 'x' does not exist..."]}

Agent action: reads error, rewrites fn body addressing the specific TypeScript error, pipes new body back via stdin

Max retries: 3 attempts. On 3rd failure: build script emits T3_FAILED with all errors. Agent reports to user with plain-English explanation and suggests removing the widget or reformulating the request.

Other widgets are not blocked — T1/T2 widgets continue generating while T3 retry is in flight.

npm install failure

Signal: PREWARM_FAILED:{"exitCode":1,"stderr":"..."}

Recovery: build script reports failure to agent with npm error excerpt. Agent surfaces to user: "Dependency install failed — this is usually a network issue or missing npm token. Run npm ci manually in [dir] to see the full error."

Prevention: build script validates GH_NPM_REGISTRY_TOKEN is set in env before starting pre-warm. Emits AUTH_MISSING:{"var":"GH_NPM_REGISTRY_TOKEN"} early if not.

UiPath auth failure

When: uip login status returns not-logged-in, or token is expired

Recovery: agent immediately runs uip login and waits for user to complete browser auth. Does not proceed to pre-warm until login is confirmed.

Build-time auth: if token expires during a long build, the dashboard's runtime auth (PKCE in browser) is independent — build-time auth is only for reading org/tenant/tenantId from login status. Token expiry during build does not break the generated dashboard.

Partial build recovery

When: build script crashes mid-flight (process kill, disk full, network drop during npm)

Detection: build script writes a .build-in-progress sentinel at start, deletes it on clean exit. If agent runs build-dashboard.mjs and sentinel exists from prior run: emits PARTIAL_BUILD_DETECTED.

Recovery options: (1) resume — skip scaffold copy + npm if already done, regenerate missing widgets only; (2) clean rebuild — delete project dir and start fresh. Agent asks user which to prefer.

Port conflict (dev server)

When: localhost:5173 already in use (another Vite app, prior dashboard)

Recovery: Vite auto-increments port (5174, 5175, …). Build script captures actual port from Vite stdout, emits SERVER_READY:{"port":5174} with actual port. Agent always reports the actual URL, never assumes 5173.

Metric hard-refuse + partial dashboards

When: user requests 4 metrics, one is in the hard-refuse table

Behaviour: agent does NOT refuse the entire dashboard. It refuses only the hard-refuse metric, explains why, and offers to build the other 3. Plan clearly marks the refused metric with the reason.

Failure mode coverage summary

✓ T3 TypeScript error → retry (max 3)
✓ npm install failure → surface + diagnose
✓ Token/auth expiry → re-login prompt
✓ Partial build crash → resume or clean
✓ Port conflict → auto-detect actual port
✓ Hard-refuse metric → partial build offer
✓ Hand-edited file on CHANGE → warn before overwrite
✓ Missing env var → early signal before pre-warm
✓ T2 descriptor schema violation → validation error before compile
