# `tests/tasks/uipath-maestro-case/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

## Scope of this folder

The `uipath-maestro-case` skill builds UiPath Case Management definitions (`caseplan.json`) from a user-supplied `sdd.md` spec, running through a six-phase workflow (Planning → Prototyping → Implementation → Validate → Debug → Publish) with explicit hard-stops at each phase boundary. The seven YAMLs in this folder cover (a) registry discovery before building, (b) a minimal "init-then-validate" smoke test, and (c) one single-node test per task-plugin type (`case-management`, `rpa`, `agent`, `api-workflow`, `process`). All five single-node tests pre-bake the full SDD into the prompt — they are coverage tests, one per plugin, that gate on `uip maestro case validate` succeeding.

## Insider markers seen in this folder

- **Skill-loading callbacks:** "Before starting, load the uipath-maestro-case skill. Read and follow its workflow steps exactly." (5 of 7 prompts) — no real customer knows to "load a skill".
- **Phase / hard-stop vocabulary:** "Phase 1 planning approval", "Phase 2 publish-for-review", "Phase 5 publish", "HARD STOP", "treat as pre-approved". These are internal workflow-control phrases from the skill's Critical Rules.
- **AskUserQuestion control plane:** "Do NOT call AskUserQuestion — it is not available in this harness" and "do NOT block on the HARD STOP / AskUserQuestion approval step". AskUserQuestion is the skill's interaction primitive, not a customer-facing concept.
- **CLI verbs leaking into user voice:** `uip maestro case validate`, `uip maestro case debug`, `uip maestro case registry pull`, `--output json` — flag-literacy that should be the agent's choice, not the customer's instruction.
- **Eval-fixture artifact contracts:** `registry_report.json` with a hard-coded JSON schema and exact field names (`cache_files_inspected`, `action_app_index_file`, `auth_state`, `cache_populated`); reference to "the cache directory at `~/.uip/case-resources/`" and `<type>-index.json` filename pattern.
- **Skill-rule callbacks:** "leave it as a skeleton task (`<UNRESOLVED>`) per the skill's rules — do NOT fabricate a taskTypeId" — the customer doesn't know the placeholder convention.
- **SDD vocabulary embedded as user request:** every single-node prompt embeds a full markdown SDD that uses internal Case-Management schema terms — `Case Identifier`, `SLA Type: Static`, `Stage Entry Conditions / WHEN / IF / Interrupting`, `exit-only`, `case-entered`, `current-stage-entered`, `required-tasks-completed`, `Component Type: AGENT`, `Task IDs` like `runRpa.runProjectEuler`, `Owner: System`, `Produced By / Consumed By` columns, `Binding: -> count`.
- **Coverage-test self-description:** each single-node SDD opens with "Single-stage, single-task case exercising the `<X>` task plugin end-to-end" — that's the test author talking to the harness, not a customer talking to the agent.
- **Harness scope-guards:** "Do NOT run `uip maestro case debug` yourself — the test harness runs it", "Build end-to-end in a single pass", "Do NOT ask for approval, confirmation, or feedback".
- **Triple project-name nesting:** "solution named `RpaSingleCase` containing a project also named `RpaSingleCase`" — eval-fixture path convenience.

## Notes for the PR description

- **All seven prompts share the same harness boilerplate trailer** ("Treat the tasks.md plan as pre-approved — do NOT block on the HARD STOP / AskUserQuestion approval step. Do NOT run `uip maestro case debug`. Before starting, load the uipath-maestro-case skill."). This is a strong signal these were written by the test author from inside the skill, not by replaying a customer transcript. Consider lifting all of the harness-control language (auto-approve, skill-loading, "Do NOT call AskUserQuestion") into the YAML's `setup` / `env` block where it belongs, so the `initial_prompt` field can hold only the user-facing ask.
- **Each single-node test embeds a 60–100-line SDD verbatim in the prompt.** The customer would not write that SDD — the *agent* would derive it during Phase 0. Two viable cleanups: (a) attach the SDD as a separate `sdd.md` fixture in the task workspace and have the prompt say "use the sdd.md I dropped in the project root" (preserves the test's coverage intent but stops conflating customer-voice with internal schema), or (b) replace the SDD with a 2–3 sentence user goal and let the agent generate the SDD via the Phase 0 interview (which is itself a skill capability worth covering).
- **The single-node SDDs self-narrate as plugin coverage** ("Single-stage, single-task case exercising the `agent` task plugin end-to-end"). That preamble is unambiguous test-author voice and should be stripped from any SDD that the prompt presents as user input — keep it in test comments / metadata if useful for grading.
- **`registry_discovery` is a legitimate keeper.** It's the only test in the folder that maps cleanly to "verify the agent uses the right CLI surface for a specific skill rule" (Rule 3 — registry pull before planning). The hard-coded report schema is an eval-grader fixture and the rest of the prompt is appropriately precise about what to inspect. Leaving it as-is is correct.
- **`init_validate` is the smoke test most likely to be confused for a customer ask.** Its first sentence is already customer-natural; rewriting it is a small, low-risk improvement that doesn't change what's being graded (a minimal Case Management project that validates).
