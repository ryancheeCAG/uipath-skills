# `tests/tasks/uipath-maestro-flow/` — prompt review

Existing test prompts vs. natural-user rewrites. Methodology in [hitl-prompts-review.html](../../hitl-prompts-review.html) and [CLAUDE.md](../../CLAUDE.md).

> **Note:** the `hitl/` subfolder of this directory is covered separately by the HITL skill review at [`../../hitl-prompts-review.html`](../../hitl-prompts-review.html) and is **not** included below. `_shared/` and `canary/` are test infra and also excluded.

## Scope of this folder

The `uipath-maestro-flow` skill is the main Author/Operate/Diagnose/Evaluate skill for `.flow` files — creating, editing, validating, debugging, publishing, running, and evaluating UiPath Maestro Flow projects via the `uip maestro flow` CLI. This folder's 50 YAML tests span eight subfolders that exercise the full surface: discovery + init smokes, single-node coverage (each pattern node, agent, RPA, API workflow, subflow, switch/decision/terminate), multi-node end-to-end builds with real connectors, structural edits (add/remove/move/group), connector-feature regression (CEQL filters, DTL fields, enums, path/query params, multiselect, required groups, searchable joins, GenerateSchema), connector-trigger filter planning, evaluator/eval-set CRUD, an authorized end-to-end eval run, and one DevCon-style holistic customer scenario.

## Insider markers seen in this folder

- **Universal harness preamble** on nearly every `single_node`, `multi_node`, and `edit` prompt: `"Validate the flow. Do NOT ask for approval, confirmation, or feedback. Do NOT pause between planning and implementation. Build the complete flow end-to-end in a single pass. Before starting, load the uipath-maestro-flow skill. Read and follow its workflow steps exactly."`
- **Runtime-variable paths in the user voice**: `=js:$vars.<triggerId>.output.documentFile`, `=js:$vars.<bt-node-id>.output`, `$vars.catalog`, `$vars.<triggerId>.output.subject`
- **`.flow` schema vocabulary**: "trigger output binding", "`triggerNodeId`", "instance JSON", "`typeVersion`", "`outputs.output.source: "=response"`", "output ports", "the wire-level node type is `uipath.pattern.deep-rag` even though the canvas display name is 'Summarize'"
- **Connector-feature internal terminology**: "DTL field whose values are loaded only on user interaction (loadByDefault=false)", "enhanced-enum order field (which exposes display labels alongside raw values)", "required-group rules", "field actions resolve correctly", "searchable join against the related object", "the path parameter on the Get Issue endpoint"
- **CLI-flag literacy**: `--output json`, `--folder-key`, `--detail`, `--type exact-match`, `--target-key greeting`, `--wait --timeout 600`, `--path`, `--inline-in-flow`
- **Eval-harness verbs and structures**: "Do NOT run flow debug", "Do NOT call `uip login`", "Do NOT call `uip solution upload`", `report.json` with fixed schemas, "use planned connector-node mode: keep the real connector node type, leave its inputs empty, validate that shape, and stop"
- **Connector registry-key gotchas**: "Connector key: use `uipath-microsoft-azureactivedirectory` exactly... display names like 'Microsoft Entra' or 'Microsoft Entra ID' are NOT registry keys"
- **Test-fixture leakage**: Orchestrator folder `Shared/uipath-maestro-flow`, connection names `is.sandboxes.test@gmail.com` and `is-sandboxes`, Slack channel `#office-bellevue`, Jira issue key `ENGCE-00000`, email `baishali13@gmail.com`, expense-approver group `expense-approvers`
- **Double-nested layout reminder repeated in customer voice**: `EvalRunDemo/EvalRunDemo/EvalRunDemo.flow`, `SmokeEval/SmokeEval/SmokeEval.flow`, "Link via `uip solution project add`" — customers don't think in layouts
- **Skill-rule callbacks**: "Per the skill's Critical Rule on upload-safety", "Read and follow its workflow steps exactly — in particular, follow the connector-trigger plugin's Step 3", "references/evaluate/references/upload-safety.md"
- **Authoring antipattern guardrails baked into the prompt**: "Use transform nodes for the data operations, not script nodes", "Transform collection inputs must reference a variable path (e.g. `$vars.catalog`), not an inline `=js:[...]` array"

## Notes for the PR description

- **The dominant fix here is removing the universal harness preamble.** 30+ prompts (essentially every `single_node`, `multi_node`, and `edit` test) end with: `"Validate the flow. Do NOT ask for approval, confirmation, or feedback. Do NOT pause between planning and implementation. Build the complete flow end-to-end in a single pass. Before starting, load the uipath-maestro-flow skill. Read and follow its workflow steps exactly."` None of that is what a customer would say. "Validate the flow" is also a skill-job, not a user-job — the skill's critical rules already mandate validation. Stripping that one paragraph turns most "Mixed" prompts into "Natural" immediately.
- **The `e2e/devcon_expense_approval` test is the gold standard** and should be cited as the model in the PR — it's vague where customers are vague ("I'm not sure exactly what all the fields should be"), uses real business roles ("manager group is 'expense-approvers'"), mocks where appropriate ("let's mock this as a script for now"), and trusts the agent to fill in the design. Most other tests should converge toward that voice.
- **The single most leaky pair of tests is `single_node/summarize` and `single_node/batch_transform`** — both embed half a `.flow` schema spec (`=js:$vars.<triggerId>.output.X`, `typeVersion: "1.0"`, `source: "=response"` not `=batchTransformResult`/`=deepRagResult`, PascalCase nested field names). If the agent passes the test it's because the prompt walked it through every gotcha; the skill's `pattern.deep-rag` and `pattern.batch-transform` plugin docs should carry that knowledge instead.
- **The `connector_features/` folder is roughly half legitimate / half fixable.** `ceql_where`, `generate_schema`, and (via its sibling) `trigger_with_filter` are bona-fide schema-shape regression tests and should keep their internal language. The DTL / enhanced-enum / required-groups / searchable-joins / field-actions tests are not — they leak connector-schema feature names into the user voice when the user would simply describe the business intent ("pick from a dropdown", "assign to several groups", "include related opportunities").
- **`evaluate/` is correctly insider-heavy on purpose.** All four tests are legitimate CLI-coverage, refusal, or fixture-driven artifact tests. No rewrites recommended there.
- **Recurring small slips worth flagging to maintainers**: (1) the `Validate the final flow file.` boilerplate at the bottom of every `connector_features/` test — duplicates a skill-level rule; (2) the Orchestrator-folder resolution paragraph (`Test connections for this task live in Orchestrator folder "Shared/uipath-maestro-flow"…`) repeated verbatim across `generate_schema`, `outlook_trigger_inbox`, `slack_channel_description`, and `slack_weather_pipeline` — that's a fixture incantation that should live in a shared test harness, not in user voice.
