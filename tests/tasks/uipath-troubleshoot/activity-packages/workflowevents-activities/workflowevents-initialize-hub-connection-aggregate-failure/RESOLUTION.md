**Root Cause:** The internal **Initialize Apps Hub Connection** activity (`UiPath.WorkflowEvents.Activities.NetCore.Activities.InitializeHubConnection`) faulted because the **Studio Web app-preview session context did not provide a valid session id / resolvable resource URL**. Its synchronous bootstrap threw `System.Activities.WorkflowApplicationException: SignalR: Invalid SessionId:  OR Orchestrator Url: `, surfaced to the job as `System.AggregateException` ("One or more errors occurred."). The agent must unwrap the `AggregateException` to this inner cause. This is a **session/URL bootstrap** failure — **not** a SignalR socket/transport failure, **not** an in-workflow error, **not** a `HandleAppRequest` null reference, and **not** an `AppRequestTrigger` connection timeout.

**What went wrong:** The last job in folder Shared — process **AppRuntimeHost** (job c0a4f7e2-1b58-4d36-9a72-8e3c5f0d2b61, started 2026-06-25 16:02 UTC, machine MOCK-HOST, Source Agent / Attended) — faulted after ~3 seconds with `System.AggregateException: One or more errors occurred. (SignalR: Invalid SessionId:  OR Orchestrator Url: )`.

**Why:** `InitializeHubConnection` runs at the start of a Studio Web app-workflow to establish the SignalR hub connection. Before opening the connection it resolves the session id and the Apps/Orchestrator resource URL; when either is blank it throws `WorkflowApplicationException("SignalR: Invalid SessionId: <id> OR Orchestrator Url: <url>")`. Here both interpolated values are empty — the session context produced no session id and no resolvable URL — so the activity threw in its **synchronous** phase, before any SignalR connection was attempted. Because the activity is async, the framework surfaced the synchronous throw as an `AggregateException` wrapping the inner `WorkflowApplicationException`. The trace `Initialize Apps Hub Connection` span confirms `SessionId = ""` (blank). The fast ~3-second fault (vs a 60-second connect wait) further confirms this is the bootstrap, not a transport timeout. Note: the actual SignalR `StartConnectionAsync` runs on a background thread that *logs and swallows* its errors, so a socket/transport failure would **not** fault the job this way — only the synchronous session/URL validation does.

**Do NOT misattribute:** This is **not** a SignalR socket / network / hub-unreachable transport failure, **not** an `AppRequestTrigger` `TimeoutException`, **not** a `HandleAppRequest` `NullReferenceException`, **not** an in-workflow bug, and **not** an Orchestrator/licensing problem. The operative cause is the **blank session id / resource URL** the activity validated and rejected.

**Immediate fix:** Re-launch the app preview / Apps run from Studio Web so a valid session id is established, and confirm the tenant's Apps (and Orchestrator) resource URL resolves for this robot. The blank values mean the session/resource-URL context was not provided to the activity — this is a session/launch-context problem, not a fix in the user's workflow.

**Preventive fix:** Launch app-workflow runs through the supported Studio Web app flow so the session id and resource URLs are populated; if the failure recurs, verify the robot can resolve the Apps/Orchestrator resource URLs (`AccessProvider.GetResourceUrl`) and obtain an access token. Raising any timeout will not help — the activity rejects the blank context immediately.
