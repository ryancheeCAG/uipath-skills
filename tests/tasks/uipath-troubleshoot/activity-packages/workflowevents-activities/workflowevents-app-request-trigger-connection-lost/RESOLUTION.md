**Root Cause:** The **SignalR connection between the UiPath App and the robot never established** within the 60-second window, so the internal **Apps Request Trigger** activity (`UiPath.WorkflowEvents.Activities.AppRequestTrigger`) faulted with `System.TimeoutException: SignalR connection did not establish within 60 seconds. Current state: Connecting`. This is a channel/transport **establishment** failure — the App↔robot link never came up — **not** an error in the user's workflow, **not** a `HandleAppRequest` null reference, and **not** an `InitializeHubConnection` bootstrap failure.

**What went wrong:** The last job in folder Shared — process **RequestListener** (job d72c5e81-3a94-4f0b-8e21-6c9a0f4b7d13, started 2026-06-25 15:10 UTC, machine MOCK-HOST, Source Agent / Attended) — faulted after ~66 seconds. The trigger waited the full SignalR establish timeout (60s) and then threw.

**Why:** `AppRequestTrigger` runs in **SignalR connection mode** and waits for the UiPath App to connect and send an invoke-workflow request over the hub. The connection never reached the `Connected` state, so `HubConnectionService.EnsureChannelConnectedAsync` threw `System.TimeoutException` after 60 seconds; the trigger's request-listener leg (`GetWorkflowRequest`) caught it and rethrew it through the activity's bookmark (`Throw` → `BookmarkWorkItem.Execute`), faulting the job. Trace evidence: the `Apps Request Trigger` span shows `ConnectionMode = SignalR` and there is **no invoke-workflow child span** — the trigger never received a request because the channel never formed. The App↔robot SignalR channel never came up, because the App instance was not connected to this run (closed / crashed / never opened) or the robot host could not reach the SignalR hub (network / proxy / firewall).

**Do NOT misattribute:** This is **not** a workflow-logic bug, **not** a `HandleAppRequest` `NullReferenceException`, **not** an `InitializeHubConnection` `AggregateException`, and **not** an Orchestrator/licensing problem. A missed *heartbeat* is handled gracefully by the trigger (it signals "connection lost" without faulting) — this fault is the **establish** path, where the connection never came up at all. The 60-second timeout is the symptom, not a value to raise.

**Immediate fix:** Ensure the UiPath App is actually open and connected to this run while the workflow waits, then re-run. Verify the robot host's connectivity to the SignalR hub — network stability, proxy, and firewall/idle-socket policies that can prevent or sever the long-lived connection. A connection that never forms means the App side never attached; raising the timeout will not help.

**Preventive fix:** Confirm the App and the robot session are reliably connected before invoking workflows, and harden the robot host's network path to the SignalR hub (stable connectivity, proxy/firewall allowances for WebSockets/long-polling). Recurring establish-timeouts on the same network path point at the proxy/firewall, not the workflow.
