# Microsoft Office 365 Activities Presentation Rules

- **Activities** — use the display name in prose (e.g., "Get Newest Email", "Move Email", "Download File", "Mark As Read/Unread", "Add Sheet"). Include the class name in parentheses on first mention so an agent can match a stack trace (e.g., "Move Email (`MoveEmailConnections`)"). Don't lead with the fully qualified class name.
- **Connections** — use the connection name as configured in Integration Service (e.g., "officeactivities"), not internal IDs. When the underlying account matters for the diagnostic, also state the authenticated Microsoft 365 user principal name.
- **Mailboxes** — distinguish the user's own mailbox from a shared mailbox or delegated mailbox
- **Mail folders** — use the user-visible folder name and parent path (e.g., "folder 'Inbox'", "folder 'Inbox/Customers/VIP'"). Don't substitute the internal Graph folder ID. For well-known folders, use the canonical Outlook display name (`Inbox`, `Sent Items`, `Drafts`, `Deleted Items`, `Archive`, `Junk Email`).
- **Email messages** — refer to a message by its `Subject` plus `ReceivedDateTime` (e.g., "the message 'Q1 invoice' received 2026-04-15 09:14"), not by message ID (only show them when the ID itself is what failed to resolve).
- **OneDrive / SharePoint items (files / folders)** — refer to a Drive item by its name with parent context (e.g., "file 'invoice.pdf' under '/Reports/2026'", "folder 'Inbox' under the 'Documents' library"). Include the item ID, URL, or drive-relative path only when the diagnostic requires it — for example, when the playbook is the 404 case and the identifier itself is what failed to resolve.
- **Drives (OneDrive vs. SharePoint document libraries)** — distinguish the user's personal OneDrive from a SharePoint document library. For SharePoint, refer to the drive by the site URL plus library display name (e.g., "library 'Documents' under site `<siteUrl>`"), not by drive ID.
- **Shared links** — refer to a sharing URL by its target (e.g., "the shared link to 'invoice.pdf'"), not by the long encoded URL. When the link itself failed to resolve, show the URL verbatim and note whether sharing is anonymous, organization-scoped, or person-scoped.
- **Excel workbooks** — refer to a workbook by its file name (e.g., "workbook 'Q1 Forecast.xlsx'"), not by the underlying drive item ID. Workbooks are OneDrive/SharePoint files, so they also follow the Drive item rule above.
- **Excel sheets (tabs inside a workbook)** — refer to a sheet by its name (e.g., "sheet 'Data'"), not by `sheetId`. When reporting a conflict on `Add Sheet` or `Rename Sheet`, show both the configured name and the existing sheet's name verbatim.
- **`ConflictBehavior` values** — refer to conflict-behavior values by the activity-facing enum display names. When recommending a value to the user, also state in plain language what it does (e.g., "set `ConflictResolution = Rename` so the new item is auto-renamed").
- **`CreateFolderIfMissing` (Move Email)** — when recommending this option, name it explicitly and state what it does: "set `CreateFolderIfMissing = true` so the destination mail folder is created if it doesn't exist". This option only exists on `MoveEmailConnections` for path-based folder input — don't suggest it for ID-based input or for other activities.
- **Local file paths** — show the full local path as configured on the activity (e.g., `C:\Reports\invoice.pdf`), not just the file name. OneDrive/SharePoint paths and local Windows paths look similar — clarify which one you mean. Forward slashes are typical for OneDrive/SharePoint paths; backslashes are typical for local Windows paths.
- **Connection scopes (delegated permissions)** — refer to scopes by their Microsoft Graph names (e.g., `Mail.Read`, `Mail.ReadWrite`, `Mail.ReadWrite.Shared`, `Files.Read.All`, `Files.ReadWrite.All`, `Sites.Read.All`, `Sites.ReadWrite.All`). When a scope is missing, name the specific scope and the operation that needs it — Graph can return 404 instead of 403 for missing-scope cases, so don't conflate the symptoms.
- **SharePoint lists / list items** — refer to a list by its display name with site context (e.g., "list 'Tickets' on site `<siteUrl>`"), not by GUID. Refer to list items by a user-visible field (e.g., title) plus the list name, not by item ID.
- **Calendars and events** — refer to a calendar by its display name (e.g., "calendar 'Team Holidays'"), not by ID. Refer to an event by its `Subject` plus `Start` time, not by event ID. When the calendar lives in a group mailbox, name the group.
- **Filter criteria (Get Newest Email, Wait For Email Received, NewEmailReceived trigger)** — when reporting a "no match" result, restate the configured filter in plain language (e.g., "messages received in the last 1 hour from `<emailAddress>` with subject containing `<subject>`"), and state the exact run timestamp. Filter-based mail activities are time-sensitive — only messages received at or before the run time are eligible.
