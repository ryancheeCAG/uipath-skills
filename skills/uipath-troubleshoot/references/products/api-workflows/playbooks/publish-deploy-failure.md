# API Workflow Pack / Publish / Deploy Fails (or Doesn't Appear in Studio Web)

## Context

What this looks like:
- `uip solution pack` / `publish` / `deploy` returns a failure, or
- Everything succeeds but the published API project is invisible / not editable in Studio Web, and a direct import fails with `Failed to import new projects at the overwrite operations`

What can cause it:
- **Wrong project type string** — packager error `No CLI tool mapping found for project type 'X'`; API workflow projects must declare `Type: "Api"` (case-sensitive).
- **Publishing the wrong artifact** — `"Invalid file type. Expected a .zip file"` when a `.nupkg` is passed instead of the wrapping `.zip` from `uip solution pack`.
- **Auth / naming** — publish 401/403 (not logged in, wrong tenant, insufficient role) or 409 name+version conflict (a package with that name+version already exists).
- **Stale generated files** — hand-committed or old-CLI `operate.json` / `package-descriptor.json` cause a descriptor mismatch; they are generated by pack and must not be committed.
- **Missing `project.uiproj`** — the on-disk API workflow project is `project.uiproj` + `Workflow.json` + `entry-points.json` (`uip api-workflow init` scaffolds exactly this). If the `.uiproj` is missing, the package still deploys but Studio Web won't open it — Studio Web requires a `project.uiproj` to recognize a folder as a project. Studio Web always writes it, so this only happens when a project was hand-assembled instead of scaffolded with `uip api-workflow init`. **Fixing the shape is an authoring task — see the `uipath-api-workflow` skill.**

What to look for:
- The exact CLI error string and stage (pack vs publish vs deploy vs Studio Web open)
- Whether the project folder has `project.uiproj` + `Workflow.json` (present for any `init`-scaffolded project)

## Investigation

1. Re-run the failing step with `--output json` and read the error: `uip solution pack <dir> <out> --output json` / `uip solution publish <zip> --output json`.
2. Confirm auth for publish/deploy: `uip login status --output json`; verify `--tenant` matches the target.
3. For the "invisible in Studio Web" case, check the project shape: does the folder contain `project.uiproj` (`ProjectType: "Api"`, `MainFile: "Workflow.json"`)?
4. For descriptor mismatches, check whether `operate.json` / `package-descriptor.json` were committed by hand.

## Resolution

- **If wrong type:** set `Type: "Api"` exactly in the project/solution manifest; re-pack.
- **If wrong artifact:** publish the `.zip` from `uip solution pack`, not the inner `.nupkg`.
- **If 401/403:** `uip login`, confirm tenant + role. **If 409:** bump `--version` and re-pack/publish.
- **If stale generated files:** delete `operate.json` + `package-descriptor.json` and re-pack (the packager regenerates them).
- **If runtime-only shape:** re-scaffold with `uip api-workflow init` and move the workflow content in, or convert in place (add `project.uiproj` + `entry-points.json`, preserve the project `Id`/`Type`). The authoring fix belongs to the `uipath-api-workflow` skill.
